# Migration

## 基本方針

1. 実行環境は `Node.js v24.14.1` を前提とする。
2. Nablarch の Java/JSP 実装は、既存の仕様だけでなく既知のバグも含めて、まずは挙動再現を優先して移行してよい。
3. 元実装に不自然さやバグがあっても、明示的な修正依頼がない限りは独自改善を入れない。

## 例外時の扱い

React、Next.js、利用ライブラリの制約で元挙動をそのまま移行できない場合は、無理にねじ込まず差分を明示する。

- 該当箇所の近くに `NOTE:` コメントを残す。
- コメントには、少なくとも次を含める。
- 元実装ではどうなっていたか。
- 何の制約で再現できないか。
- 現在実装ではどう扱ったか。

例:

```ts
// NOTE: 元の JSP では blur 時に即時バリデーションしていたが、利用中ライブラリの制約で同一タイミング再現が難しい。
// NOTE: 現在は submit 時バリデーションで代替している。
```

## 相談が必要なケース

次のどれかに当てはまる場合は、実装を強行する前にユーザーへ相談する。

1. 大きな回避実装や特殊ケース分岐が必要になる。
2. React やライブラリの制約を超えるために、不自然な状態管理や DOM 操作が必要になる。
3. 元画面との差分が UX や業務フローに影響しそう。
4. 再現よりも設計変更に近い判断になっている。

## 判断の優先順位

1. 元実装の挙動再現
2. 制約がある場合の差分明記
3. 無理筋になりそうなら早めに相談
